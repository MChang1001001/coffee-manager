# 咖啡豆管理系统数据库草案 v1

## 1. 设计目标

第一版数据库设计目标是：

1. 支撑个人咖啡豆档案管理。
2. 支撑咖啡豆包装封面上传。
3. 支撑结构化评价和风味标签。
4. 支撑每次实际冲煮记录。
5. 后续可以平滑扩展到多用户、附件表、批次表、统计分析和冲煮模板。

第一版采用 MySQL 关系型数据库为主。

Redis 暂时只作为预留能力，可用于后续登录态、缓存或验证码，不作为第一阶段核心依赖。

---

## 2. 核心表清单

第一版建议包含以下表：

```text
users
coffee_beans
coffee_reviews
brew_records
flavor_tags
coffee_bean_flavor_tags
review_flavor_tags
```

说明：

- `users`：用户表。
- `coffee_beans`：咖啡豆主表，一条记录代表一包实际购买的豆子。
- `coffee_reviews`：咖啡豆评价表。
- `brew_records`：冲煮记录表。
- `flavor_tags`：风味标签表。
- `coffee_bean_flavor_tags`：咖啡豆与风味标签关联表。
- `review_flavor_tags`：评价与风味标签关联表。

---

## 3. ER 关系概览

```text
users 1 ─── n coffee_beans
users 1 ─── n coffee_reviews
users 1 ─── n brew_records

coffee_beans 1 ─── n coffee_reviews
coffee_beans 1 ─── n brew_records

coffee_beans n ─── n flavor_tags
coffee_reviews n ─── n flavor_tags
```

简化理解：

- 一个用户可以创建多条咖啡豆记录。
- 一个咖啡豆可以有多条评价。
- 一个咖啡豆可以有多条冲煮记录。
- 一个咖啡豆可以有多个风味标签。
- 一个评价也可以绑定多个风味标签。

---

## 4. 通用字段约定

除关联表外，主要业务表建议包含以下通用字段：

```text
id
user_id
created_at
updated_at
deleted
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 主键 ID |
| user_id | BIGINT | 数据归属用户 ID |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |
| deleted | TINYINT | 逻辑删除标记，0 未删除，1 已删除 |

第一版建议使用逻辑删除，方便误删恢复和后续统计。

## 5. users 用户表

### 5.1 表用途

用于保存登录用户信息。

第一版虽然以个人单用户为主，但仍保留用户表，方便后续扩展。

### 5.2 建表草案

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
    username VARCHAR(64) NOT NULL COMMENT '用户名',
    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希',
    nickname VARCHAR(64) DEFAULT NULL COMMENT '昵称',
    avatar_url VARCHAR(500) DEFAULT NULL COMMENT '头像URL',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',
    UNIQUE KEY uk_users_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

### 5.3 字段说明

| 字段 | 说明 |
|---|---|
| username | 登录用户名 |
| password_hash | 密码哈希，不保存明文密码 |
| nickname | 展示昵称 |
| avatar_url | 用户头像，第一版可不用 |

## 6. coffee_beans 咖啡豆表

### 6.1 表用途

用于保存咖啡豆档案。

第一版中，一条 `coffee_beans` 记录代表一包实际购买的咖啡豆。

### 6.2 建表草案

```sql
CREATE TABLE coffee_beans (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '咖啡豆ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',

    name VARCHAR(128) NOT NULL COMMENT '咖啡豆名称',
    origin VARCHAR(128) DEFAULT NULL COMMENT '产地',
    region VARCHAR(128) DEFAULT NULL COMMENT '具体产区',
    farm VARCHAR(128) DEFAULT NULL COMMENT '庄园/农场',
    variety VARCHAR(128) DEFAULT NULL COMMENT '品种',
    process_method VARCHAR(64) DEFAULT NULL COMMENT '处理法',
    roast_level VARCHAR(64) DEFAULT NULL COMMENT '烘焙度',
    roaster VARCHAR(128) DEFAULT NULL COMMENT '烘焙商',

    roast_date DATE DEFAULT NULL COMMENT '烘焙日期',
    purchase_date DATE DEFAULT NULL COMMENT '购买日期',
    open_date DATE DEFAULT NULL COMMENT '开封日期',
    finish_date DATE DEFAULT NULL COMMENT '喝完日期',

    net_weight_grams DECIMAL(8,2) DEFAULT NULL COMMENT '净含量，单位g',
    price DECIMAL(10,2) DEFAULT NULL COMMENT '购买价格',
    currency VARCHAR(16) DEFAULT 'CNY' COMMENT '币种',

    status VARCHAR(64) NOT NULL DEFAULT 'UNOPENED' COMMENT '状态',

    cover_image_url VARCHAR(500) DEFAULT NULL COMMENT '包装封面图片访问URL',
    cover_image_object_key VARCHAR(500) DEFAULT NULL COMMENT '包装封面文件存储Key',

    overall_rating DECIMAL(3,1) DEFAULT NULL COMMENT '综合评分缓存，1-5分',
    review_count INT NOT NULL DEFAULT 0 COMMENT '评价数量缓存',
    brew_count INT NOT NULL DEFAULT 0 COMMENT '冲煮记录数量缓存',

    notes TEXT DEFAULT NULL COMMENT '备注',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',

    KEY idx_coffee_beans_user_id (user_id),
    KEY idx_coffee_beans_status (status),
    KEY idx_coffee_beans_roast_date (roast_date),
    KEY idx_coffee_beans_purchase_date (purchase_date),
    KEY idx_coffee_beans_overall_rating (overall_rating),
    KEY idx_coffee_beans_deleted (deleted)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='咖啡豆表';
```

### 6.3 关键字段说明

| 字段 | 说明 |
|---|---|
| name | 咖啡豆名称，必填 |
| origin | 国家或大产地，例如 Ethiopia、Colombia |
| region | 更具体产区，例如 Yirgacheffe |
| farm | 庄园或农场 |
| variety | 咖啡品种，例如 Heirloom、Gesha、Caturra |
| process_method | 处理法 |
| roast_level | 烘焙度 |
| roaster | 烘焙商 |
| status | 当前状态 |
| cover_image_url | 前端用于展示的封面图 URL |
| cover_image_object_key | 后端用于删除/迁移文件的存储 key |
| overall_rating | 综合评分缓存，便于列表排序 |
| review_count | 评价数量缓存 |
| brew_count | 冲煮记录数量缓存 |

### 6.4 状态枚举

建议后端枚举值：

```text
UNOPENED              未开封
OPENED                已开封
FINISHED              已喝完
DISCARDED             已废弃
REPURCHASE_CANDIDATE  回购候选
```

## 7. coffee_reviews 咖啡豆评价表

### 7.1 表用途

用于保存用户对某包咖啡豆的评价。

一包豆子可以有多条评价，例如不同阶段、不同冲煮经验后的评价。

### 7.2 建表草案

```sql
CREATE TABLE coffee_reviews (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '评价ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    coffee_bean_id BIGINT NOT NULL COMMENT '咖啡豆ID',

    overall_rating DECIMAL(3,1) NOT NULL COMMENT '综合评分，1-5分',
    aroma_rating DECIMAL(3,1) DEFAULT NULL COMMENT '香气评分',
    acidity_rating DECIMAL(3,1) DEFAULT NULL COMMENT '酸度评分',
    sweetness_rating DECIMAL(3,1) DEFAULT NULL COMMENT '甜感评分',
    bitterness_rating DECIMAL(3,1) DEFAULT NULL COMMENT '苦感评分',
    body_rating DECIMAL(3,1) DEFAULT NULL COMMENT '醇厚度评分',
    aftertaste_rating DECIMAL(3,1) DEFAULT NULL COMMENT '余韵评分',

    content TEXT DEFAULT NULL COMMENT '文字评价',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',

    KEY idx_reviews_user_id (user_id),
    KEY idx_reviews_coffee_bean_id (coffee_bean_id),
    KEY idx_reviews_overall_rating (overall_rating),
    KEY idx_reviews_deleted (deleted)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='咖啡豆评价表';
```

### 7.3 评分约束

业务层建议限制：

```text
评分范围：1.0 - 5.0
评分刻度：允许 0.5
```

数据库层可以先不强制 `CHECK`，由后端参数校验处理，兼容性更好。

## 8. brew_records 冲煮记录表

### 8.1 表用途

用于保存每一次实际冲煮日志。

一条记录代表一次真实冲煮，而不是模板。

### 8.2 建表草案

```sql
CREATE TABLE brew_records (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '冲煮记录ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    coffee_bean_id BIGINT NOT NULL COMMENT '咖啡豆ID',

    brew_method VARCHAR(64) NOT NULL COMMENT '冲煮方式',
    bean_amount_grams DECIMAL(6,2) DEFAULT NULL COMMENT '粉量，单位g',
    water_amount_ml DECIMAL(7,2) DEFAULT NULL COMMENT '水量，单位ml',
    ratio VARCHAR(32) DEFAULT NULL COMMENT '粉水比，例如 1:15',
    water_temperature DECIMAL(5,2) DEFAULT NULL COMMENT '水温，单位℃',
    grind_size VARCHAR(128) DEFAULT NULL COMMENT '研磨度',
    brew_time_seconds INT DEFAULT NULL COMMENT '冲煮总时长，单位秒',

    result_summary VARCHAR(255) DEFAULT NULL COMMENT '结果摘要',
    result_notes TEXT DEFAULT NULL COMMENT '结果反馈',
    is_recommended TINYINT NOT NULL DEFAULT 0 COMMENT '是否推荐：0否，1是',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',

    KEY idx_brew_records_user_id (user_id),
    KEY idx_brew_records_coffee_bean_id (coffee_bean_id),
    KEY idx_brew_records_brew_method (brew_method),
    KEY idx_brew_records_is_recommended (is_recommended),
    KEY idx_brew_records_deleted (deleted)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='冲煮记录表';
```

### 8.3 冲煮方式枚举

建议后端枚举值：

```text
V60
ORIGAMI
KALITA
CHEMEX
FRENCH_PRESS
AEROPRESS
ESPRESSO
MOKA_POT
COLD_BREW
OTHER
```

### 8.4 推荐记录规则

第一版允许同一咖啡豆存在多条推荐冲煮记录。

原因：

- 同一豆子可能同时适合 V60 和 Espresso。
- 不同冲煮方式下可能都有推荐参数。
- 第一版不强制唯一性，体验更自由。

## 12. 是否需要外键

第一版建议：

> 数据库层暂不强制外键，业务层保证关联正确。

原因：

1. 开发和调试更灵活。
2. 逻辑删除场景下外键处理更简单。
3. 后续如果需要严格约束，可以再补充外键。

但在代码层必须保证：

- 创建评价时，`coffee_bean_id` 必须属于当前用户。
- 创建冲煮记录时，`coffee_bean_id` 必须属于当前用户。
- 修改咖啡豆、评价、冲煮记录时必须校验 `user_id`。
- 删除咖啡豆时，需要同步逻辑删除关联评价和冲煮记录，或限制删除。

## 14. 字段取舍说明

### 14.1 为什么不单独建 batch 表

第一版中，一条咖啡豆记录已经代表一包实际购买的豆子。

所以暂时不需要额外增加 `coffee_bean_batches`。

后续如果需要区分：

- 同一豆子的基础信息
- 不同购买批次
- 不同烘焙批次
- 不同库存状态

可以再拆成：

```text
coffee_beans
coffee_bean_batches
```

### 14.2 为什么包装封面直接放在 coffee_beans 表

第一版只支持一张包装封面，不支持多附件。

因此直接使用：

```text
cover_image_url
cover_image_object_key
```

比单独建附件表更简单。

后续如果支持多图、冲煮照片、文件管理，再拆出：

```text
attachments
```

### 14.3 为什么保留 rating 缓存字段

`coffee_beans.overall_rating`、`review_count`、`brew_count` 是列表页常用展示字段。

直接缓存可以减少列表查询时的聚合压力。

第一版可以在新增、编辑、删除评价和冲煮记录时同步更新。

## 17. 后续迁移方向

当项目进入第二阶段或需要更强能力时，可以考虑：

1. 新增 `attachments` 表
   - 支持多图。
   - 支持不同业务对象的附件。
   - 支持文件大小、类型、原始文件名等元数据。

2. 新增 `coffee_bean_batches` 表
   - 区分同款豆子的不同购买/烘焙批次。

3. 新增 `brewing_devices` 表
   - 管理滤杯、磨豆机、咖啡机等器具。

4. 新增 `recipe_templates` 表
   - 将优秀冲煮记录沉淀为模板。

5. 新增 `purchase_records` 表
   - 管理购买渠道、价格变动、回购记录。

## 18. 第一版数据库结论

第一版数据库设计保持轻量但完整。

核心策略：

1. 咖啡豆即一包实际购买的豆子。
2. 咖啡豆、评价、冲煮记录分表存储。
3. 风味标签采用多对多。
4. 包装封面先直接放在咖啡豆表。
5. 保留 `user_id`，但不做复杂权限。
6. 使用逻辑删除。
7. 用缓存字段优化列表展示。

这套设计足够支撑第一版 MVP，也方便后续平滑扩展。
